# EzLang 文档与实现缺口补全计划

## P0 — 实现文档规定的语法解析与中间代码生成
- [x] 为文档中的 `parallel { ... }` 代码块添加解析器支持
  依据：`docs/doc.md` 中定义了 `parallel` 语法，但 `grammar/EzLang.g4` 未定义对应标识符与语法规则。
  工作内容：补充词法标识符与语法规则，重新生成 `compiler/src/parser*` 下的解析器产物；新增语义流程与挂起状态跟踪逻辑；完成中间代码生成、运行时应用程序二进制接口占位实现；补充解析、语义分析、代码生成全流程测试。
  验收标准：`const p = parallel { return 1; };` 这类代码可正常解析、语义分析并生成中间代码；若在不支持的目标平台使用，需给出明确的编译期错误提示。

- [x] 为变量锁前缀 `rp let`、`wp let` 添加解析与语义校验支持
  依据：`docs/doc.md` 定义了 `rp let` / `wp let` 语法，但当前语法规则仅支持 `(let | const | static)` 声明变量。
  工作内容：新增 `RP` / `WP` 词法标识符；在符号表中记录锁策略；生成运行时锁元数据与锁操作指令；实现读优先、写优先、默认排序三类锁逻辑。
  验收标准：`rp let cache = []`、`wp let queue = []` 可正常解析，要么编译为完整的运行时锁调用逻辑，要么抛出精准的「功能暂不支持」诊断信息。

- [x] 统一文档与语法文件中的类型、指针语法
  依据：文档使用 `*I8` 指针类型与 `Blob(...)` 写法，语法文件暂无指针类型定义，且结构体构造语法仅对内置结构体生效。
  工作内容：修改文档与接口声明，统一为当前已支持的 `Blob`/`Str` 写法；补充类型解析测试用例。
  验收标准：`docs/doc.md`、`docs/stdlib.md` 中所有类型表达式均可正常解析，或文档已修正为可解析语法。

- [x] 统一文档与语法文件中的结构体方法声明语法
  依据：文档中结构体方法写法为 `add(this: Date, ...) => Void`，但语法文件仅支持 `名称 = 函数字面量` 形式。
  工作内容：支持无函数体的方法签名声明；补充内置风格声明的测试用例。
  验收标准：内置类型 `Date`、`Error`、`Blob`、网络连接方法、界面示例代码均能匹配解析器语法。

- [x] `permission.camera`、`gravity.left`、`contentMode.scaleToFill` 这类带点号的常量声明，替换为现有写法
  工作内容：选定命名空间/常量语法，实现解析、语义分析、代码生成全链路支持；或修改包文档，改用符合语法的结构体/模块写法。
  验收标准：`docs/ez-web-ui.md`、`docs/ez-android-ui.md`、`docs/ez-ios-ui.md` 中所有常量代码片段均可正常解析。

- [x] 支持文档定义的字典字面量键名语法与动态键类型结构
  依据：文档支持字符串键 `{ "Content-Type" = ... }` 与动态键类型 `{ [key: Str]: Str }`，但语法文件仅允许普通标识符作为字典键；代码生成逻辑仅使用简易追加式字段名表。
  工作内容：增加字符串键、表达式键解析能力；完善动态键语义兼容逻辑；实现字典键查询、更新、删除语义。
  验收标准：请求头相关示例、`Dict<K,V>` 接口可正常解析、类型校验，并完成键的读写查询。

- [x] 支持无初始化器的变量声明，或同步修改文档
  依据：文档示例 `let arr: User[]?;` 未设置初始值，但语法文件要求变量声明必须带赋值表达式。
  工作内容：`let arr: User[]?;` 这类允许无初始化的变量初始化为空值。
  验收标准：文档中所有变量声明示例均可正常解析与类型校验。

## P1 — 完善 `docs/doc.md` 定义的语言语义
- [x] 完善 flow 语法的中间代码生成，实现调度语义
  依据：文档规定 flow 需转换为状态机，支持依赖等待、执行挂起、任务取消、竞争逻辑；当前代码仅插入运行时标记，`sleep`/`race` 做特殊处理，其余逻辑均同步执行。
  工作内容：定义任务、future、挂起源对应的中间代码与运行时接口；将语义依赖关系对接至代码生成模块；重构为原生运行时代码。
  验收标准：flow 内部多个独立挂起操作可并发执行；依赖读取会等待前置任务完成；副作用执行完毕后 flow 才正常退出；超时、取消事件可正常传递，且覆盖对应测试用例。

- [x] 修正 race 语法，对齐文档接口与运行行为
  依据：文档定义 `race(pl = [() => ...], timeout = ...)`，但当前代码仅支持 `race(任务编号: 整型, 超时时间: 整型)` 格式，调用运行时接口参数不匹配。
  工作内容：支持可调用对象数组参数；实现分支任务创建、首个结果返回、终止其余任务、可选超时、异常传递逻辑。
  验收标准：文档中 `race(pl=[...])` 示例可正常解析、类型校验、生成代码，且运行结果符合预期。

- [x] 实现 `throw` / `catch` 异常值传递逻辑
  依据：当前 `catch` 固定返回空值；`throw` 仅做局部跳转与标记，无法传递 `Error` 异常对象，也无顶层异常终止逻辑。
  工作内容：定义异常数据传输接口、`catch` 返回值类型；实现异常在函数、flow、parallel 语法中的传递；补充未捕获异常的终止处理逻辑。
  验收标准：`const err = catch { throw Error(...) }` 可获取完整可访问的异常对象；未捕获的异常会终止程序并输出诊断信息。

- [x] 按文档实现 `typeof` 运算符，返回类型标识与元数据（而非固定数值 0）
  依据：语义分析模块将 `typeof` 判定为字符串类型，文档要求其返回类型标识并支持运行时类型判断，当前代码生成逻辑固定输出常量 0。
  工作内容：区分编译期与运行时 `typeof` 语义；为结构体、基础类型分配唯一标识；实现文档中的类型对比、位运算判断逻辑；同步修正文档类型说明。
  验收标准：`typeof 42`、`typeof User[]`、`typeof err & Error == Error` 类型合法，且生成的中间代码逻辑正确。

- [x] 完善可选类型与联合类型的运行时行为
  依据：当前可选类型、联合类型基于结构体实现，但 `?` 语法语义混乱，空值表示不统一，`expr?` 解包功能不完善，联合类型匹配、类型校验逻辑缺失。
  工作内容：定义空值字面量/构造方法；实现安全解包、强制解包；补充类型标签校验、联合类型构造、模式匹配与类型判断逻辑。
  验收标准：可选值、`parseInt` 等解析类接口、`a?.text()` 链式调用、联合类型分支行为均符合文档定义。

- [x] 完善偏函数应用与柯里化闭包功能
  依据：文档规定 `?` 占位符用于生成闭包；当前代码虽支持调用处占位符，但独立占位符返回固定零值，闭包整体逻辑需全链路验证。
  工作内容：确定闭包内存布局、捕获变量规则、函数指针调用逻辑、内存池生命周期管理；完善剩余参数的类型校验。
  验收标准：`let addTwo = fn(a = 2, b = ?); addTwo(b = 3)` 可正常执行并返回 5，通过代码生成与运行时测试。

- [x] 统一值语义、`this` 引用语义与聚合类型拷贝规则
  依据：文档规定普通变量默认值拷贝、`this` 为引用传递；当前代码对聚合类型有时传指针、有时传值，语义混乱。
  工作内容：制定聚合类型参数、返回值的应用二进制接口；强制 `this` 以指针形式传递；为结构体、数组、字典补充内存拷贝、值拷贝测试。
  验收标准：修改 `this` 可正常作用于调用方；普通赋值执行值拷贝；函数调用不会意外造成聚合类型别名引用。

- [x] 完善结构体/类型结构的鸭子类型与泛型类型推导
  依据：文档支持鸭子类型、动态结构、泛型结构体/方法推导、局部类型推导；当前语义分析仅完成部分校验，代码生成仅对显式泛型函数做单态化处理。
  工作内容：实现结构兼容性判断、结构字段默认值；完善调用处、构造方法的泛型参数自动推导；支持泛型结构体与泛型方法。
  验收标准：文档中结构类型、键值对、`identity(42)` 自动推导、结构体扩展示例均可通过语义分析与代码生成测试。

- [x] 完善标记语法的中间代码生成逻辑
  依据：文档规定标记语法会转换为带属性、子节点的函数调用/构造方法；当前代码仅返回标签名字符串。
  工作内容：将标记语法转换为标准调用/构造形式，保留属性与子节点；对目标工厂函数做类型校验。
  验收标准：`<text color="blue">...</text>` 生成的中间代码、运行行为与文档描述的等价调用完全一致。

- [x] 修正内存池（Arena）行为，对齐文档的作用域与生命周期规则
  依据：文档规定内存池支持按作用域保存/恢复、跨作用域值拷贝、线程独占、变量重赋值复用；当前仅使用全局 1MB 内存池，无溢出处理，代码块提前返回时不执行内存恢复。
  工作内容：所有代码退出分支均补充作用域内存回收；新增内存溢出诊断与扩容策略；实现返回聚合类型时的父作用域拷贝；创建线程本地内存池；定义变量重赋值后的内存分配规则。
  验收标准：提前返回、嵌套代码块内存回滚、跨作用域聚合值返回、大内存分配失败/扩容、flow/parallel 内存隔离均通过测试。

- [x] 统一数组（List）内存布局，同步文档或修改代码以适配分页存储结构
  依据：文档定义 `Type[]` 布局为 `{ 长度: 64位整型, 容量: 64位整型, 数据指针: 类型指针 }`；当前代码采用分页存储结构 `{ 分页指针数组, 长度, 容量, 分页数量 }`。
  工作内容：选定一套内存布局，同步更新文档、应用二进制接口、标准库集合实现；保证数组索引、标准库接口统一使用该布局。
  验收标准：文档、代码生成、标准库声明、运行时工具函数的数组布局完全一致。

- [x] 按文档补全所有运算符语义
  依据：文档明确了有符号/无符号整除、取余、右移，向量标量广播、向量比较、复合赋值单次求值、逻辑短路规则；当前代码以有符号整型运算为主，向量运算功能有限，复合赋值逻辑简陋。

## 建议执行顺序
1. 稳定语法与文档示例：修复解析器漏洞、重新生成解析器产物、上线文档示例编译校验。
2. 完善命令行构建链路：实现标准库、原生封装、外部库的链接能力，保证可执行测试用例正常运行。
3. 落地核心运行时语义：补全内存池、异常捕获抛出、可选/联合类型、`typeof`、flow/race/parallel、读写锁逻辑。
4. 按依赖顺序完善标准库：统一内存库错误码 → IO → 文件系统 → 时间 → 系统 → 格式化 → 网络 → 集合。
5. 完善包解析逻辑，依次实现网页界面、安卓界面、iOS 界面包。
6. 全量整改文档与接口一致性，将签名校验、示例校验长期纳入持续集成流程。
